## 一、场景描述

### 1.1 一个具体的决策时刻

九月初，选课系统开放的第三个晚上。

林晓是一所政法大学法学专业的大三学生，绩点卡在保研边缘。下学期的《民事诉讼法》是必修课，但有四位老师同时开班——选谁，她有选择权，也几乎只有这一次机会：错过退选窗口，这门课的成绩就会跟着她进入保研排名。

教务系统里，四个教学班的全部信息是：课程名、老师姓名、学分、上课时间、剩余名额。没有一个字回答她真正想问的问题：哪位老师给分压不压？点不点名？期末是闭卷还是论文？上课会不会点人起来回答——她社恐，这一条对她是真正的雷点，但从来没有人把它写进任何一份"攻略"。

她做了所有能做的事：私聊了两位上过这门课的学姐，一位说"X 老师还行"，另一位说"听说他这两年变严了"；翻了校园墙和小红书，搜出十几条零散帖子，有夸有骂，最新的一条停在两年前；又找到一张在朋友圈流传的"红黑榜"截图——上面恰好没有这四位老师中的两位。查分工具？她的学校没有。两个晚上过去，她手里攒下一堆互相打架的碎片，每一条都"仅供参考"，没有人敢把话说死。

退选截止前一小时，她在两个名字之间来回切换，最后凭感觉点了提交。

这不是个例。我们访谈了四所学校、三种选课机制下的学生，听到的是同一种困境。

> 注：本场景为**复合画像**，素材取自 4 场真实访谈（受访者 A（研究生）——退选窗口与学分约束情节；受访者 B（大四·留学冲均分）；受访者 C（研一）——"说法互相打架、没人敢定论"情节；受访者 D（大四·法学·保研）——"课堂点人"维度与"必修选老师"情节来自此场），细节均有访谈出处，详见交付物1《场景调研记录与需求分析》。

### 1.2 谁在经历这个时刻：高 stakes 选课者

不是所有学生都痛。访谈中最重要的跨场洞察是：**痛感与课程 stakes 成正比**。随便选、可补救的水课，学生凭感觉选、踩了雷也无所谓；但当一门课关乎 GPA、保研、留学申请、毕业——选错代价高、退课窗口短、名额与学分约束强——学生才真的投入、真的怕错。

因此本产品的目标用户收窄为**高 stakes 选课者**：保研/留学/考研冲刺者、绩点敏感者、研究生，以及法学这类"必修课也要选老师"的大专业学生。我们不试图让所有学生都觉得选课是个大问题——它对很多人确实不是。

### 1.3 真正卡住他们的是什么

直觉的答案是"费时间"，但访谈否定了这个答案。高 stakes 用户其实**愿意**花时间：一位受访者为一次选课连续三天、每天约两小时、问了十多个人。真正的问题是——**投入买不来确定性**。时间花完，她得到的仍然是：

- **散**：信息散落在私聊、群文件、校园墙、小红书，没有一处是全的；
- **乱**：每个人口径不同（"给分还行"到底是多少分？），好评差评混杂排列，要自己数比例、自己判断前后是否矛盾——"损耗精力"；
- **旧**：老师会换、大纲会变、给分风格会变，两年前的评价参考价值不明——"像抽奖"；
- **虚**：学长学姐不敢担责、刻意留余地，"搞得我更犹豫"；
- **不对题**：所有评价都是"对所有人说的"，没有一条回答"**对我**——一个要保研、社恐、怕点名的人——这门课合不合适"。

一句话：**卡点不是找不到信息，而是判断不了、比较不了。**

### 1.4 评课社区解决了什么、没解决什么

这个卡点的前半段（散、旧）其实已有被验证的解法：像 USTC 评课社区这样的成熟平台，把评价集中到一处、绑定到具体老师和学期，"散"和"旧"被大幅缓解。

但即使在评课社区最成熟的学校，后半段依然原样卡着：评价仍是自由文本，**读和判断的负担还在学生身上**；评分和总结是对所有人统一的，"**对我合不合适**"仍然无人回答。而更多的学校（如场景中林晓的学校）连前半段都没有——一片信息荒漠。

所以本产品的做法是分两层：**先搭一个评课社区作为内容底座**——把散落的点评集中、结构化、绑定到老师×学期，解决"散、旧"，这一层也是冷启动入口；**再在它之上叠一层 AI 决策层**——语义结构化 + 个人画像匹配 + 可解释推荐，解决"判断不了、对我合不合适"。前者是地基，后者才是差异化与 AI 价值所在。一句话：**做评课社区，但不止于评课社区**——前台形态是社区，主体价值在其上的决策层（这一形态分层在第三节展开）。

### 1.5 一个关键设计约束：痛的时刻与用的时刻是错开的

踩雷的痛发生在学期中——"想退课，但名额和学分不够了"（一位受访者被选课币的沉没成本卡住，忍了一整个学期）；而使用产品的时刻在下一次选课周，中间隔着几个月，痛感早已被时间磨平。

这意味着**产品不能指望"痛"来驱动使用**：必须把使用摩擦压到接近零——零评价时也有用、查询零门槛、写评价近乎零填写负担——让不那么痛的人也顺手用。这条约束将贯穿第三节的全部功能设计。

### 1.6 收束

选课是一个**高 stakes、信息不对称、近乎不可逆**的决策，而学生此刻的决策装备，是一堆散、乱、旧、虚、不对题的碎片。把这些碎片变成"对我而言可解释的决策依据"——这就是本产品的起点。下一节展开它带来的用户价值。

## 二、用户价值陈述

第一节描述了卡点：选课是一个高 stakes、信息不对称、近乎不可逆的决策，而学生此刻的决策装备是一堆散、乱、旧、虚、不对题的碎片。本节回答：把这些碎片变成"对我而言可解释的决策依据"，到底为用户创造了什么价值。

### 2.1 三个痛点，三层价值

我们不堆砌价值点，只认调研里真实出现、且有受访者自发佐证的三层。它们逐层递进：先让信息**可比较**，再让判断**对我适配**，最后让结论**可信**。

| 真实痛点（来自访谈） | 产品怎么解决 | 创造的价值 |
|---|---|---|
| 评价散落、口语化、口径不同，**没法横向比较**——B 要自己"数哪个比例占得多"、判断好评差评是否矛盾，"损耗精力" | AI 低摩擦地把口语评价抽成统一的多维 schema（给分/难度/作业量/点名/课堂互动……） | **可比较 → 判断成本下降**：几门课、几位老师一眼可比，不必再逐条读、逐条脑补归纳 |
| 评价是"对所有人说的"，**同一门课对不同人价值相反**——B 直言"直接工作/考研的人对均分不敏感，出国为目的均分是首要"；D 社恐怕被点起来回答（课堂点人），而"给建议的人常不觉得这是雷点" | 按个人目标画像（留学/保研/考研/就业/求通过）+ 情境（必修 vs 选修、学分高低）做匹配，并给出"为什么适合你/为什么避开"的解释 | **个性化决策 → 从"看评价"升级到"对我而言的建议"**：这是 AI 带来的真增量，也是竞品都没做的一层 |
| 旧评价像抽奖、给分没有客观依据、没人敢下定论——C"像抽奖""学长不敢把话说死"，D"无法把握'高'是多少分" | 评价绑定"这位老师·这学期"并标新鲜度；叠加客观数据（给分分布/选满率）与主观口碑交叉验证 | **不确定性下降 → 可信度上升**：让"凭感觉和运气"变成"有据可依" |

三层背后是同一件事：**受访者已经在用最原始的方式做这三件事**——B 手动数好评比例、还主动提"完全可以把好评差评中评分类汇总"；C 自发说"需要一个让大家从不同维度评分的东西，对一门课更公平更综合"；D 自己列 Word 表格排序、做老师红黑榜。产品不是无中生有地创造需求，而是把学生**已经在费力手搓的动作**自动化、规模化。

### 2.2 一句话价值

> 选课是一个**高 stakes、信息不对称、近乎不可逆**的决策：一门课关乎 GPA、保研、留学，而退选窗口一过、名额一满就改不了——访谈中真有人想退却被选课币和名额卡住、忍了一整个学期。本产品把散落、口语化的群体经验，在近乎零填写摩擦下结构化成可比较、可个性化的数据，再叠加客观数据，让高 stakes 选课者从"凭感觉和运气"，变成"**对我而言可解释的决策**"。

注意价值口径的两个刻意取舍：

- **不卖"省时间"。** 高 stakes 用户本就**愿意**花时间（B 为一次选课花了约 6 小时、问 10 多个人），他们的痛不在耗时，而在**投入买不来确定性**——时间花完仍判断不了。所以价值是"降低判断成本与不确定性"，不是"几小时压缩到几分钟"。
- **不替学生做决定。** 产品给的是"可解释的依据 + 对你的建议"，最终选择权留给学生。这既符合真实决策心理（没人愿意把保研押在一个黑箱推荐上），也守住了 AI 的可信边界。

### 2.3 回到题目：AI 在这里"提了什么效"

题目是"AI 提效大作业"，所以必须正面回答：这里的 AI 到底提了什么效？答案不是"用 AI 写了段总结"，而是解决了一个**结构性障碍——群体经验无法被规模化利用**。散落在无数学生脑子里、用各种说法表达的主观感受，过去只能靠一次次私聊、一张张手抄表格低效流转；AI 把它们一次性转成机器可处理、可比较、可个性化推送的数据资产。具体提了三个"效"：

| 提效维度 | 没有 AI 时 | 有 AI 后 |
|---|---|---|
| **信息聚合效率** | 学生手动翻几十条零散评价，自己脑补归纳 | AI 自动把口语评价归纳成结构化画像 |
| **信息可比性** | 每条评价各说各话，无法横向比较 | 统一维度结构化，几门课一眼可比 |
| **决策个性化** | 评价"对所有人说的"，学生自己换算"对我意味着什么" | 算法按个人画像匹配，AI 翻译成"为什么适合你" |

而这套"提效"有一条清醒的边界，恰恰是它经得起追问的地方：**只有"把人话变成数据"和"把结果讲成人话"这两步用 AI，其余的匹配、排序、筛选、统计一律用确定性算法**。因为后者规则清晰、可枚举，用算法又快、又便宜、又可复现、又可解释；让大模型去算匹配分反而不稳定、不可控。能主动说清"哪些用了 AI、哪些故意不用 AI"，本身就是这个方案对"AI 提效"四个字最诚实的回答。这条 AI / 算法的分工，将在第三节的功能架构里具体落地。

## 三、功能体系与设计思路

前两节确立了"做什么、为什么有价值"。本节回答"怎么做"——但顺序上，我们先讲**怎么裁的**（设计原则 + 裁决方法），再讲**裁出了什么**（功能体系 + 架构），最后讲**故意不做什么**（砍除清单）。这样安排，是因为这个产品最该被看见的不是功能多，而是**取舍清醒**。

### 3.1 设计原则：两层形态 + 一条主线 + 两条约束

**先定形态：产品分两层**（承接 §1.4「做评课社区，但不止于评课社区」）——

- **基础社区层（内容底座）**：一个评课社区——点评流、课程库、课程详情、多维评分、AI 总结、同课对比、写评价。**所有人可用、不登录可看**，是数据燃料的来源，也是冷启动入口。形态上它和成熟评课社区无异，是产品的"地基"。
- **决策增强层（差异化与 AI 价值）**：**登录绑定教务后才出现**——我的选课工作台、个性化匹配、可解释推荐、ChatBI 问数。它把社区沉淀的群体经验，结合个人画像与教务约束，变成"对我而言的决策"。这一层才是产品的主体价值。
- 一条铁律贯穿两层：**个性化只发生在决策层**——公共社区层（课程库列表）只做统一排序（评分高/点评多/最新），不做"对我"的匹配；"对我合不合适"只出现在登录后的工作台、课程详情和 ChatBI。

**再定三句约束，每个功能都挂得上：**

- **主线 · AI/算法分工**：AI 把人话变成数据（语义结构化），算法用数据做匹配排序（确定性计算），AI 再把结果讲回人话（解释与问数）。**AI 只在语义层的两端，中间的计算一律算法。**
- **约束一 · 摩擦近零**：第一节已论证"痛的时刻与用的时刻错开"，产品不能靠痛驱动 → 必须做到**零评价也有用、查询零门槛、写评价近零负担**。这条直接决定了"社区底座先行、客观数据先有用"和"AI 低摩擦抽取"两个关键选择。
- **约束二 · 克制**：判断功能去留只看**用户价值**，不看"竞品有没有做过"——竞品做过的刚需（AI 总结、同课对比）照做、做扎实，竞品没做的也不硬凑差异点；同时确定性任务坚决不用 LLM。能做的不一定都做，避免功能膨胀稀释主线。

### 3.2 功能怎么裁的：四问裁决法

功能清单不是头脑风暴堆出来的，每一个候选都过了同一套四问，任何一问不过就降级或砍掉：

1. **用户价值有证据吗？**（访谈/实验里有没有真实信号支撑，还是凭空想象）
2. **竞品做过的，是刚需还是伪差异化？**（竞品做没做**不是否决理由**——是用户刚需就照做、做扎实〔如 AI 总结、同课对比〕，只是不吹成我们的差异化；只有"做了也没人真需要"的才砍）
3. **该用 AI 还是算法？**（语义理解才上 AI，确定性计算用算法）
4. **零评价时就有用，还是依赖评价飞轮？**（决定它属于"第一天就能用的地基"还是"成熟期增值层"）

### 3.3 三层架构：AI 两端，算法居中

产品的数据流是一条清晰的流水线——口语评价从左端进，决策视图从右端出，中间三层各司其职，底下托一层"零评价也有用"的客观数据：

```
   口语评价(输入)
        │
        ▼
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ ① AI 结构化层     │ ──▶ │ ② 算法匹配层      │ ──▶ │ ③ 解释生成层      │ ──▶ 决策视图
│ (schema v2 抽取)  │     │ (画像×权重         │     │ · "为什么适合你/   │     (呈现/输出)
│  〔AI〕           │     │  排序/筛选/比较)   │     │   避开"〔AI 翻译，  │
│                  │     │  〔确定性算法〕    │     │   锚定字段可追溯〕  │
│                  │     │                  │     │ · ChatBI 问数入口  │
└──────────────────┘     └──────────────────┘     │   (问题→查询)〔AI〕 │
        ▲                          ▲               └──────────────────┘
        │                          │
   写评价(近零摩擦)         ┌──────────────────────────┐
                          │ 客观数据载体层            │
                          │ 选满率/给分分布/课表冲突/  │
                          │ 学分进度  〔确定性算法〕    │
                          │ —— 零评价也有用，与主观    │
                          │    评价交叉验证            │
                          └──────────────────────────┘

   贯穿全程：时效性（每条评价绑定"这位老师 × 这个学期" + 信息新鲜度标注）
```

**读这张图的关键**：标〔AI〕的是**两端**——① 把人话抽成数据（评价结构化），③ 把数据讲回人话（F4 解释"为什么适合你/避开" + ChatBI 把自然语言问题翻成查询意图）。其中 F4 解释虽由 AI 生成，但**每条理由都锚定算法实际用到的结构化字段、客观数据与权重贡献**，可追溯、不凭空发挥——既"讲人话"又不黑箱。中间 ② 的匹配/排序/筛选，以及底层客观数据的统计，全是确定性算法。这正是 2.3 那条"AI/算法分工"的落地。

### 3.4 功能体系（按 §3.1 两层形态组织）

> 说明：P0 按"基础社区层 / 决策增强层"两层呈现（产品形态视角）；§3.3 的"AI 结构化→算法匹配→解释"三层是数据处理流，与此正交——同一个功能在形态上属某层，在数据流上各归其位。

#### P0 · 基础社区层（评课社区底座 · 不登录可看 · demo 必做）

| 功能 | 说明 | AI/算法 | 为什么是 P0 |
|---|---|---|---|
| **点评流** | 首页实时滚动最新点评（"XX 用户点评了《XX 课》XX 老师 · X 分钟前" + 评分/标签摘要，可点进课程详情） | 算法 | 让新用户一眼看到"社区是活的、评价在持续更新"；同时承载"邀请你评价已修课"的供给入口，是冷启动的关键发动机 |

| **F10 AI 总结（分板块）** | 把一门课几十条评价归纳成分板块摘要（整体/适合谁/难度/给分/是否推荐） | **AI** | 高 stakes 同学"快速读懂一门课"的刚需——竞品做过但仍是刚需，做扎实；不当差异化 |
| **F11 同课多老师横向对比** | 同名课不同老师并排比多维字段 + 客观给分 | 算法 | "选老师"是核心场景（含必修选老师）；并列基础上叠加决策层 F3 个性化重排，才是增量 |
| **F5 时效性** | 每条评价绑定"老师×学期"；默认只显当前老师评价、旧的折叠标灰；标信息新鲜度 | 算法 | 直击"像抽奖"痛点；成本极低、收益高 |
| **写评价入口 + 轻量结构化模板** | 近零摩擦写一条评价：结构化打分 + 一句自由文本，AI 自动抽标签 | AI+算法 | 社区底座的 UGC 来源；产品面写这层，冷启动/飞轮机制整块归交付物3 |

| 收获 | 高 / 中 / 低 / 未提及 | 展示 + 匹配 | AI 抽取 |
| 点名·考勤 | 严 / 偶尔 / 无 / 未提及 | 展示 + 匹配 | AI 抽取 |
| 老师风格 | 自由短标签（不含给分） | 展示 | AI 抽取 |

#### P0 · 决策增强层（选课助手 · 登录绑定教务后出现 · demo 必做）

| 功能 | 说明 | AI/算法 | 为什么是 P0 |
|---|---|---|---|
| **教务绑定登录** | 绑定教务账号 → 同步课表 / 成绩 / 已修课程 / 本学期可选课程（demo 为模拟绑定，不真连教务） | 算法 | 决策层入口，带来"零评价也有用"的个人客观数据。合规边界：用历史/已授权数据，不实时抓取余量 |
| **载体 A · 我的选课工作台** | 整合本学期可选课程、课表、时间冲突、学分进度，并把每门课的评价/客观数据/AI 解释汇到一处 | 算法 | "选课决策助手"主场；高频留存载体，也是唯一被验证且可复制的冷启动引流路径；学分进度戳中访谈 A"判断毕业学分差多少" |
| **F3 个性化匹配层** | 按目标画像(留学/保研/考研/就业/求通过)+情境(必修vs选修、学分高低)给"适合/不适合你"与排序。**只在工作台 / 课程详情"对我是否合适" / ChatBI 出现，不进公共课程库列表** | **算法** | 差异化主轴——竞品全缺。MVP 只做"标签筛选 + 预设权重模板"，不用 LLM |
| **F4 可解释推荐** | 把算法的匹配结果翻成自然语言"为什么适合你/为什么避开"，并点明依据的字段/数据 | **AI**（翻译解释） | 把匹配分变成看得懂、可追问的理由；AI 负责"讲人话"，但内容锚定算法给的结构化依据，不凭空生成（既可解释又不黑箱） |
| **F8 ChatBI 自然语言问数** | 右下角悬浮入口（小 logo）→ 对话框自然语言提问（"高数哪个老师讲得好且给分高""通识里给分高又轻松的课"） | **AI**（问题→查询）+ 算法（查/算） | 把"同课选老师 + 众课筛目标"变成一句话就能问的入口；AI 只翻译意图，计算与答案由算法在结构化数据上完成、附证据 |

> 两层之间的关系：基础社区层**不登录就能用**（浏览点评、查课程、看客观给分、读 AI 总结），保证"零评价也有用、第一天就有价值"；决策增强层**登录绑定教务后解锁**，把社区内容 + 个人画像 + 教务约束合成"对我的决策"。这正是 §1.4「做评课社区、但不止于评课社区」的功能落地。

### 3.5 明确砍除清单：我们故意不做什么

敢于砍，比堆功能更能说明想清楚了。砍的标准**不是"竞品做过"**（AI 总结、同课对比竞品做过，我们照样做），而是**这件事服不服务"个人选课决策"、踩不踩合规红线**：

| 砍除项 | 为什么砍 |
|---|---|
| **X1 站点级数据可视化** | 站点级统计图表不服务个人选课决策；我们只保留"面向个人的决策视图"（USTC 的 Highcharts 是这方面的成熟参照，不重复） |
| **X2 把"排序更聪明"当卖点** | 全局排序的小样本偏差 USTC 用贝叶斯收缩已解决；我们的排序价值在"按你的画像个性化重排"，不在"算得更准"——这点只作护栏、不当卖点 |
| **X3 实时教务监控 / 抓取余量** | 合规风险高、加重学校系统压力，且决策更需要历史抢手程度而非实时余量 |
| **X4 资料 / 课件分享** | 与个人选课决策主线无关，挪出 MVP 保持聚焦 |

这张清单的意义不止于"省事"：它划清了护城河——**评课社区底座我们照做、做扎实（它是地基与冷启动入口）；但真正的增量在其上的个性化决策层（画像匹配 + 可解释 + ChatBI）。我们不在已经饱和的地方（站点级可视化、全局排序算法）跟竞品内卷，把力气全压在竞品都没做的决策层上。** 这也直接引出下一节的竞品对比。

## 四、竞品对比分析

第三节已经埋下竞品对比的底色：评课社区底座我们照做、做扎实，但增量全押在其上的"个性化决策层"。本节正面回答——**相比现有产品，我们到底差异在哪、这个差异稳不稳。**

### 4.1 一张图看清定位：我们在右上角

把市面上的选课/评课产品放进两个维度，差异一目了然：

- **横轴 · 信息形态**：从"自由文本 / 平均分"（人读、不可计算）→ "结构化、可计算"
- **纵轴 · 决策方式**：从"统一评价"（对所有人一个答案）→ "个性化匹配"（对你一个答案）

```
              个性化匹配 ▲
                        │
                        │                  ┌─────────────┐
                        │                  │  本产品      │
                        │                  │ 结构化+画像  │
                        │                  │   匹配       │
                        │                  └─────────────┘
                        │
   ─────────────────────┼─────────────────────────────▶
   自由文本/平均分        │                    结构化可计算
                        │
        RMP ·           │   USTC（多维评分+AI总结+归一化排序）
   THU（推荐指数+        │   UST SPACE（4维+结构化考核清单
     满屏自由文本）       │              +Timetable Planner）
                        │   321CQU / Berkeleytime（客观成绩分布）
                统一评价 │
```

> 注：RMP = RateMyProfessors，北美高校通用的教授评价平台，"纯自由文本 + 统一评价"的代表。

竞品几乎全部落在下半区——无论信息结构化到什么程度，**给所有人的都是同一个答案**。右上角"结构化 + 个性化匹配"这个象限是空的，这就是本产品的位置。

### 4.2 基线 vs 差异：哪些不能当卖点，哪些才是真增量

强竞品已经把一批功能做成了**行业基线**。我们的态度：基线里属于**用户刚需**的（AI 总结、同课对比）照做、做扎实，但**不靠它们做差异化**——诚实区分"基础功能"和"差异点"，是这个方案不被"一问就穿"的前提：

| 已是基线（不当差异化卖点） | 谁做到了 |
|---|---|
| 多维结构化评分 | USTC（4维）、UST SPACE（4维 + 字母等级） |
| AI 总结 | USTC（分板块）；非全行业标配，但已不能当差异化 |
| 同课多老师比较 | USTC |
| 排行榜 + 归一化排序（贝叶斯收缩处理小样本） | USTC |
| 站点级数据可视化 | USTC（Highcharts） |
| 结构化考核清单 | UST SPACE（9 项打勾） |
| 客观成绩分布 | 321CQU、Berkeleytime |

而把三家最成熟的评课社区（USTC / UST SPACE / THU）逐项拆开后，有四件事**三家全缺**——这才是我们的真增量：

| 真差异（三家深拆竞品全缺） | 我们怎么做 |
|---|---|
| **个性化匹配**（按个人目标 + 情境给不同答案，而非给所有人同一个排名） | F3 画像 × 预设权重模板 |
| **个人决策解释**（为什么这门课适合你 / 避开，且能追问） | F4 可解释推荐 + F8 ChatBI 问数 |

一句话概括差异：**竞品回答"这门课大家怎么看"，我们回答"这门课对我是不是合适"。**

### 4.3 收敛：差异稳，但命门在冷启动

三家深拆竞品收敛出三条结论：

- **个性化 = 三家全无** → 我们最稳的差异化主轴，在两个最强竞品面前同时成立。
- **结构化 schema 各家深浅不一** → 我们做超集（13 个 AI 抽取字段，含竞品都没有的"课堂互动·点人"），并借鉴 UST SPACE 的结构化考核清单。
- **AI 总结已被头部竞品做出**（USTC 分板块，UST SPACE/THU 无）→ 非全行业标配，是"快速读懂一门课"的刚需，我们照做（升 P0）但不当差异化卖点。

但竞品对比也照出了这个赛道真正的命门，必须诚实写明：**所有人的难点都不是功能，而是冷启动。** 连最成功的 USTC 运营 11 年，多数课程仍只有 1–2 条评价、评分严重正偏；顶校 THU 的评课社区课程覆盖率仅约 3.5%（截至 2026-06 抓样）。这说明评课内容飞轮极难启动——而这正是为什么本产品把"零评价也有用的客观数据载体"放进 P0：**先用不依赖评价的工具撑住第一天的价值与留存，为评价飞轮买时间**（冷启动的完整两段设计属于交付物3 运营方案）。

### 4.4 双轨定位：面对两类学校，姿态不同

最后点一笔落地策略（展开在交付物3）。本产品面对两类学校，进入姿态不同：

- **已有强评课社区的学校**（如 HKUST 的 UST SPACE、USTC）："散、旧"已被社区解决，我们的增量集中在**个性化匹配 + AI 语义结构化 + 可解释决策**，形态更像社区之上的**工具层 / 增值插件**，可对接已有评价内容。
- **没有评课社区的学校**（如访谈中的某信息荒漠校——公开评课内容近乎为零）：要先解决**内容供给**（客观数据载体先行 + 评价飞轮起步），决策层才有燃料。

**内容社区（数据燃料）是前提，AI 选课决策助手是重点，两者必须兼顾。** 这与第三节"客观数据载体先行"的设计一脉相承，也是这个产品区别于"又一个评课社区"的根本所在——它不止是又一处看评价的地方，而是把群体经验变成"对我而言可解释的决策"的工具。
